# Contract: Auth API changes

**Feature**: specs/010-credential-encryption | Base path: `/api/auth`

Session mechanics (opaque token, HttpOnly cookie, `requireSession`) are **unchanged**.
Only payloads change, plus one new public endpoint.

---

## NEW — GET `/api/auth/salt/:username`

Public. Returns the KDF salt the client needs before it can derive keys for login.

**Response `200`** (always — including unknown usernames):

```json
{ "kdfSalt": "<base64url 22 chars>" }
```

- Known username → the user's real stored `kdfSalt`.
- Unknown username → **deterministic decoy**: base64url(first 16 bytes of
  SHA-256(lowercased username + `SALT_DECOY_PEPPER` env value)). Same username always
  gets the same decoy, so the endpoint cannot be used to enumerate accounts (timing and
  shape identical either way; login then fails with the same generic 401 as today).
- Username is lowercased before lookup (matches login semantics).
- No rate-limit change (login throttling still applies at `POST /login`).

---

## CHANGED — POST `/api/auth/register`

**Request** (was `{ username, displayName, password }`):

```json
{
  "username": "alice1",
  "displayName": "Alice",
  "authHash": "<base64url 43 chars — HKDF-auth of PBKDF2(password, kdfSalt)>",
  "kdfSalt": "<base64url 22 chars — generated by the client>",
  "vaultKeyWrapped": "v1.wk.<iv>.<ct>"
}
```

- Password policy (3–10 chars) is enforced **client-side only** now — the server never
  sees the password. Server validates `authHash`/`kdfSalt`/`vaultKeyWrapped` formats.
- Server hashes `authHash` with the existing PBKDF2 storage scheme → `passwordHash`.

**Response `201`** (was `{ user }`):

```json
{ "user": { "id": "...", "username": "alice1", "displayName": "Alice" },
  "vaultKeyWrapped": "v1.wk.<iv>.<ct>" }
```

Errors unchanged: `400 validation_error`, `409 username_taken`.

---

## CHANGED — POST `/api/auth/login`

**Request** (was `{ username, password }`):

```json
{ "username": "alice1", "authHash": "<base64url 43 chars>" }
```

**Response `200`**:

```json
{ "user": { ... }, "vaultKeyWrapped": "v1.wk.<iv>.<ct>" }
```

Errors unchanged: `401 invalid_credentials` (generic), `403 account_disabled`,
`429 too_many_attempts`.

---

## CHANGED — GET `/api/auth/me`

**Response `200`**: `{ "user": { ... }, "vaultKeyWrapped": "v1.wk.<iv>.<ct>" }`

(`vaultKeyWrapped` is returned so a refreshed tab can re-unlock with only the password —
the locked-vault flow — without a full re-login.)

---

## UNCHANGED — POST `/api/auth/logout`

No payload change. Client-side contract addition: the SPA MUST zero its in-memory vault
key when logout resolves (and on any 401-driven session reset).

---

## Security invariants

1. No request or response body ever contains: the password, the master key, the wrap
   key, or the unwrapped vault key.
2. `authHash` is useless for decryption (HKDF domain separation) and is re-hashed
   server-side before storage (a DB dump is not login-replayable).
3. Salt endpoint responses are indistinguishable between existing and non-existing
   accounts.
